# 规划设计院合同管理系统 - 总体设计方案

## 1. 核心功能模块清单

系统的核心目标是实现规划设计院合同的“全生命周期管理”，打破部门间的信息壁垒，提升流转与审批效率。

1. **工作台 (Dashboard)**
   - 个人待办事项（待审批、待归档等）
   - 数据概览（本月新签合同额、累计收款额、逾期未收预警）
   - 快捷入口（起草合同、客户登记、立项申请）
2. **客户与项目管理**
   - **客户库**：客户基本信息、对接人、历史合作记录、信用评级。
   - **项目库**：项目立项信息、项目负责人(PM)、项目进度状态、关联的合同池。
3. **合同全生命周期管理 (核心)**
   - **合同起草/录入**：支持标准模板起草或直接上传己方/对方草拟的合同附件。
   - **合同审批流**：部门负责人 -> 法务 -> 财务 -> 院领导（根据金额或类型动态流转）。
   - **合同台账**：合同编号生成、基本信息（名称、金额、签订方、负责人、履行期限）。
   - **变更/补充协议**：处理合同标的、金额、期限的变更审批及归档。
4. **财务与履约管理**
   - **收款计划**：根据合同约定节点（如预付款、进度款、尾款）制定计划。
   - **开票申请**：基于合同及收款记录发起开票申请及审批。
   - **认款/核销**：财务确认到账后，与具体合同及收款计划进行核销关联。
5. **归档与附件管理**
   - **电子归档**：用印后的扫描件、附件（技术任务书、报价单等）集中存储。
   - **纸质归档位置追踪**：记录纸质合同存放在财务室或档案室的具体位置。
6. **统计报表**
   - 部门/项目合同业绩统计。
   - 收付款执行率分析。
   - 逾期款项明细表。
7. **系统设置(后台管理)**
   - 组织架构（部门、岗位）与用户管理。
   - 角色与权限分配 (RBAC)。
   - 审批流程配置（字典、节点）。
   - 基础数据字典（合同类型、开票归类等）。

## 2. 用户角色与权限建议

基于 RBAC 模型，满足规划设计院典型职能划分。

| 角色/岗位名称 | 核心职责与数据权限 |
| :--- | :--- |
| **业务人员/项目经理 (PM)** | 录入客户/项目/合同草稿，发起审批，发起开票申请。**权限**：仅查看/操作自己作为负责人的客户和合同数据。 |
| **部门负责人/所长** | 审核本部门员工提交的合同和开票申请，查看本部门业绩。**权限**：查看和审批本部门所有数据。 |
| **法务/风控专员** | 核心职责是审查合同条款风险，处理合同变更。**权限**：查看所有待审合同详情，驳回或修改建议。 |
| **财务专员** | 审核业务条款（金额、付款方式），处理开票、登记收款记录（认款）。**权限**：全局跨部门查看合同财务信息，管理票据和资金流水。 |
| **档案管理员** | 负责最终纸质原件和电子扫描件的归档核对与封存。**权限**：查看所有已生效合同，维护归档记录。 |
| **院领导/高管** | 终审重大合同，查看全院宏观经营数据面板。**权限**：全局数据查看权限，无具体录入权限。 |
| **系统管理员 (Admin)** | 维护基础字典、人员账号变更、审批流节点调整。**权限**：系统设置模块的全部权限，但不干预业务数据审批。 |

## 3. 合同全生命周期业务流程

1. **【前期准备】**：PM 录入客户信息 -> 发起内部“项目立项”（如有）。
2. **【合同起草】**：PM 登录系统，关联客户和项目，录入拟签合同的关键要素（标的、金额、期限、付款节点），并上传合同文本（Word/PDF）。
3. **【内部会审】**（审批流）：
   - *一级审批*：部门负责人（确认业务合理性）。
   - *二级审批*：法务专员（审查条款风险）。
   - *三级审批*：财务专员（审查付款条件和税务合规性）。
   - *特批节点*：若金额超过设定阈值（如50万），触发院领导审批。
4. **【用印登记】**：审批通过后，生成统一合同流水号，PM 携线下或线上凭单去办公室盖章。
5. **【生效与履约】**：
   - 合同状态变为“履行中”。
   - PM 根据约定的节点发起**开票申请** -> 财务审批并开具发票（登记发票号）。
   - 客户打款后，财务进行**收款登记**，系统自动核销对应的收款计划，更新已收金额/未收金额。
6. **【归档变更】**：
   - 签订后需将双方盖章的终版 PDF 上传作为**电子归档**，并登记纸质原件去向。
   - 若履约中发生变更，需发起“补充协议签署”，走类似审批并挂载于主合同下。
7. **【完结】**：所有款项收齐，项目交付物通过，合同标记为“已结案”。

## 4. 核心页面结构

使用常见的中后台两栏或三栏布局设计：

- **/ (Dashboard)**
  - 工作台：待办列表、数据卡片、快捷入口。
- **/customer (客户管理)**
  - /customer/list (客户台账：列表、搜索、新建)
  - /customer/detail/[id] (客户详情：含基本信息、关联项目、关联合同)
- **/project (项目管理)**
  - /project/list (立项列表)
- **/contract (合同管理核心)**
  - **/contract/draft** (草拟录入/发起流程，多步表单设计：1.基本信息 -> 2.收款节点 -> 3.附件)
  - **/contract/ledger** (合同台账，提供强大综合查询：按部门、金额、状态、签约期筛选)
  - **/contract/detail/[id]** (【重型页面】合同详情页：分标签页展示：概览总览、审批轨迹记录、开票与收款明细列表、附件及电子归档、补充协议关联)
- **/finance (财务履约)**
  - **/finance/payment-plan** (全院收款计划跟踪表，超期预警标红)
  - **/finance/invoice** (开票申请列表与开票登记)
  - **/finance/receipt** (收款认领与凭证入账记录)
- **/system (系统设置)**
  - /system/users (用户与部门维护)
  - /system/roles (角色权限树分配)
  - /system/dicts (数据字典：如下拉框选项配置)

## 5. 数据库表设计建议 (核心实体)

基于关系型数据库 (如 PostgreSQL 或 MySQL)，使用软删除 (`is_deleted`) 和通用审计字段 (`created_at`, `updated_at`, `created_by`)。

| 表名 (Entity) | 说明 | 核心字段示例 |
| :--- | :--- | :--- |
| `sys_user`, `sys_dept`, `sys_role` | RBAC 基础表 | user_id, dept_id, role_id... |
| `biz_customer` | 客户档案 | id, name, type (企企/政府), contact_person, phone, tax_number, address |
| `biz_project` | 项目立项 | id, customer_id, name, pm_id (项目负责人), status, budget |
| `biz_contract` | 合同主表 | id, project_id, customer_id, dict_contract_type, <br>`serial_no` (合同编号), `name`, `total_amount`, <br>`sign_date`, `start_date`, `end_date`, `pm_id`, <br>`status` (草稿/审批中/履行中/已归档/结案), `archive_location` |
| `biz_contract_payment_plan` | 收款/付款计划 | id, contract_id, phase (如期数：一期), expected_amount, expected_date, condition (收款前提), actual_received_amount, status |
| `biz_invoice` | 开票记录 | id, contract_id, plan_id(可选), amount, invoice_code, invoice_number, apply_by, apply_date, status |
| `biz_receipt` | 收款/资金流水 | id, contract_id, invoice_id(关联发票), amount, receipt_date, operator_id, remark |
| `biz_contract_file` | 附件/归档表 | id, contract_id, file_name, file_url, file_type (草案/终稿/扫描件) |
| `sys_workflow_instance` <br>(或外接工作流引擎) | 审批流实例 | id, biz_type (合同/开票), biz_id, current_node, status |
| `sys_workflow_log` | 审批记录 | id, instance_id, node_name, approver_id, action (同意/驳回), comment, create_time |

## 6. 模块之间的数据关系

- **客户 (Customer)** 是一切业务的上游源头 `1:N` 关联 **项目 (Project)** 与 **合同 (Contract)**。
- 规划设计院场景下，一个大型开发项目可能包含多个维度的专业合同（如：规划设计合同、专项审查合同、后期咨询合同），因此 **Project** `1:N` **Contract**。
- **Contract 主表** 具有最高权重，向下放射出多个从属实体集合：
  - `1:N` 关联 **Payment Plan (收款计划)** 定义了预期何时收多少钱。
  - `1:N` 关联 **Invoice (开票记录)** 追踪已开出的税务发票。
  - `1:N` 关联 **Receipt (收款记录)** 追踪实际到账资金，这些金额汇总(`SUM`)决定了 Contract 的已收金额与欠款。
  - `1:N` 关联 **File (附件)**。

## 7. 推荐开发顺序

遵循“由底层到上层，由基础到业务，再由主线到分支”的商用系统开发法则：

- **阶段 一：基础设施与基座 (Sprint 1)**
  - 搭建前后端项目骨架 (Next.js)。
  - 实现登录、鉴权校验 (JWT / Session)。
  - 实现 RBAC 基础（部门、人员、角色管理页面）。
  - 系统公共组件封装（带分页筛选的高级表格、通用弹窗、文件上传组件）。
- **阶段 二：前置业务档案 (Sprint 2)**
  - 数据字典管理模块。
  - 客户库管理 (CRUD 及联系人维护)。
  - 项目立项管理模块。
- **阶段 三：合同主核心引擎 (Sprint 3 - 最关键部分)**
  - 实现简单的状态机或接入工作流引擎。
  - 合同起草表单（联动客户和项目下拉选择）。
  - 主台账列表（实现复杂查询和数据权限过滤——只能看自己或本部门的）。
  - 合同详情聚合页（Overview页）。
- **阶段 四：财务与履约分支 (Sprint 4)**
  - 在合同详情页内开发“收款计划”子模块。
  - 开票申请表单与审批流。
  - 收款认款流水登记功能。
  - 开发定时任务或聚合查询处理“逾期应收账款预警”。
- **阶段 五：仪表盘与交付 (Sprint 5)**
  - 实现 Dashboard 各类统计图表与业绩排行。
  - 合同电子归档校验。
  - 系统综合测试、权限联调与部署。
